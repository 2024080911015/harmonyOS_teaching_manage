# 教学过程管理系统项目逻辑说明

## 1. 项目定位

本项目是一个基于 Flask 和 SQLite 的教学过程管理 REST API 服务端，主要面向以下四类用户：

- `STUDENT`：学生
- `TEACHER`：教师
- `ACADEMIC_STAFF`：教务人员
- `ADMIN`：管理员

项目目前只提供 JSON API，不包含前端页面。主要业务包括用户认证、个人资料、课程选课、课表查询、通知发布、教室预约、学生请假、教师调课和后台基础数据管理。

## 2. 总体架构

项目采用 Flask 应用工厂和 Blueprint 分模块组织代码：

```text
客户端请求
    ↓
app.py 创建 Flask 应用并注册 Blueprint
    ↓
auth_guard.py 验证 Token 和用户角色
    ↓
blueprints/ 中的业务模块处理请求
    ↓
services.py 执行课表计算、时间冲突检查等公共业务
    ↓
db.py 获取当前请求的 SQLite 连接
    ↓
data/teaching.db
```

### 2.1 入口文件

`app.py` 中的 `create_app()` 负责：

1. 创建 Flask 应用；
2. 加载 `config.py` 中的配置；
3. 注册数据库连接的关闭逻辑；
4. 注册各业务 Blueprint；
5. 添加跨域响应头；
6. 提供 `/api/health` 健康检查接口。

直接执行 `python app.py` 时，服务默认监听：

```text
http://0.0.0.0:8080
```

当前直接启动方式启用了 Flask `debug` 模式，适合本地开发，不适合直接用于生产环境。

### 2.2 Blueprint 模块

| 模块 | 主要职责 |
| --- | --- |
| `blueprints/auth.py` | 登录、退出、当前用户、修改密码 |
| `blueprints/profile.py` | 个人资料、角色专属资料、首页统计 |
| `blueprints/common.py` | 院系、班级、学期、教室等基础查询 |
| `blueprints/notifications.py` | 通知发布、接收、已读和撤回 |
| `blueprints/courses.py` | 课程查询、选课、退课和课表 |
| `blueprints/bookings.py` | 教室预约申请和审批 |
| `blueprints/leaves.py` | 学生请假申请和审批 |
| `blueprints/schedule_changes.py` | 教师调课申请和审批 |
| `blueprints/admin.py` | 用户、院系、班级、教室、学期、课程和排课管理 |

## 3. 配置和数据库

### 3.1 配置

`config.py` 定义了以下主要配置：

- 数据库默认位置：`data/teaching.db`；
- Token 默认有效期：7 天；
- 请求体最大大小：8 MB；
- 可以使用环境变量 `TEACHING_DB` 指定其他数据库文件；
- 可以使用环境变量 `TOKEN_EXPIRE_DAYS` 修改 Token 有效期。

### 3.2 数据库连接

`db.py` 使用 Flask 的 `g` 对象保存当前请求的数据库连接：

1. 当前请求第一次访问数据库时建立连接；
2. SQLite 查询结果使用 `sqlite3.Row`，可以按字段名读取；
3. 开启外键约束；
4. 设置 15 秒忙等待时间；
5. 请求结束后自动关闭连接。

写操作通常使用显式事务：

```text
BEGIN IMMEDIATE
    ↓
执行一组关联写入
    ↓
成功：commit
失败：rollback
```

这样可以避免申请状态、审批记录和操作日志只写入一部分。

### 3.3 数据表分组

数据库中的主要表可以分为以下几组：

- 用户与认证：`sys_user`、`auth_token`；
- 角色资料：`student_profile`、`teacher_profile`、`academic_staff_profile`；
- 基础数据：`department`、`class_group`、`classroom`、`semester`；
- 教学数据：`course`、`course_schedule`、`course_enrollment`；
- 教室预约：`room_booking`、`room_booking_log`；
- 学生请假：`leave_request`、`leave_request_course`；
- 教师调课：`schedule_change`；
- 通知：`notification`、`notification_recipient`；
- 审批审计：`approval_record`。

应用启动时不会自动执行数据库迁移。首次部署或数据库结构发生变化后，需要单独运行：

```powershell
python migrate.py
```

`migrate.py` 会补充学期、请假课程关联、教室预约日志、通知截止时间、索引和选课校验触发器等结构，并且可以重复执行。

## 4. 认证与权限

### 4.1 登录流程

登录接口为：

```http
POST /api/auth/login
```

处理流程如下：

```text
提交账号和密码
    ↓
查询 sys_user
    ↓
校验密码哈希和账号状态
    ↓
生成随机 Token
    ↓
写入 auth_token 和过期时间
    ↓
返回 Token、用户信息和角色
```

除登录接口外，请求通常需要携带：

```http
Authorization: Bearer <token>
Content-Type: application/json
```

### 4.2 Token 校验

`auth_guard.py` 中的 `auth_required` 会检查：

- 请求是否携带 Bearer Token；
- Token 是否存在；
- Token 是否已撤销；
- Token 是否过期；
- 对应账号是否仍然启用。

验证通过后，当前用户和当前 Token 会分别保存到：

```python
g.current_user
g.current_token
```

`roles_required(...)` 在身份认证基础上继续检查角色。例如：

```python
@roles_required("STUDENT")
```

表示接口只能由学生调用。

退出登录会撤销当前 Token。修改密码成功后，系统会撤销该用户除当前 Token 以外的其他 Token。

## 5. 各角色的业务逻辑

### 5.1 学生

学生的主要使用流程为：

```text
登录
 → 查看课程
 → 选课或退课
 → 查看个人课表
 → 提交教室预约
 → 提交课程请假
 → 查看通知
```

#### 选课

学生选课时，系统会检查：

- 课程是否存在并启用；
- 课程是否开放选课；
- 当前时间是否在选课起止时间内；
- 课程是否达到容量上限；
- 学生是否已经选择该课程；
- 新课程是否和已有课程产生时间冲突。

课程冲突同时考虑：

- 学期；
- 起止周；
- 单周、双周或每周；
- 星期；
- 起止节次。

部分规则由后端和数据库触发器双重校验。选课或退课后，数据库触发器会同步维护课程的已选人数。

退课不会直接删除选课记录，而是修改记录状态并保留历史。

#### 请假

学生只能为自己当前已选的课程提交请假。一次请假可以关联多门课程，关联关系保存在 `leave_request_course` 中。

请假状态流转为：

```text
PENDING
 ├─ APPROVED
 ├─ REJECTED
 └─ CANCELLED
```

只有 `PENDING` 状态的请假可以由学生取消。涉及相关课程的教师可以查看请假，但审批操作由教务人员或管理员完成。

### 5.2 教师

教师的主要使用流程为：

```text
登录
 → 查看授课课程、学生和课表
 → 查看涉及本人课程的学生请假
 → 发布通知
 → 提交教室预约
 → 提交调课申请
```

#### 调课

调课支持三种类型：

- `TIME`：只调整时间；
- `ROOM`：只调整教室；
- `BOTH`：同时调整时间和教室。

提交调课申请时，系统会检查：

- 原排课是否属于当前教师；
- 新教室是否存在并可用；
- 新周次是否位于原课程周次范围内；
- 是否已有相同范围的待审批或已通过申请；
- 教师在新时段是否有其他课程；
- 教室在新时段是否被课程占用；
- 教室在新时段是否已有预约。

调课提交时检查一次冲突，审批通过前还会再次检查，避免审批期间数据变化导致冲突。

### 5.3 教务人员和管理员

教务人员和管理员主要负责：

- 创建、查询和维护用户；
- 启用或停用用户；
- 重置用户密码；
- 管理院系、班级和教室；
- 管理学期、课程和排课；
- 控制课程状态和选课开放状态；
- 审批教室预约、学生请假和教师调课；
- 管理通知和查看全局待办。

教务人员只能创建学生和教师账号，管理员可以创建所有角色的账号。项目不提供公开注册接口，账号需要预先创建。

新增或修改排课时会检查：

- 对应教师和教室是否合法；
- 教师在相同时间是否已有课程；
- 教室在相同时间是否已有课程；
- 单双周和周次范围是否真正重叠；
- 对应学期是否存在；
- 教室在课程时间是否已有预约。

## 6. 三条审批流程

### 6.1 教室预约

学生和教师都可以申请教室：

```text
PENDING
 ├─ 审批通过 → APPROVED → 过期批处理 → FINISHED
 ├─ 审批驳回 → REJECTED
 └─ 申请人取消 → CANCELLED
```

提交申请前会检查：

- 预约日期不能早于当前日期；
- 教室必须处于可用状态；
- 参与人数不能超过教室容量；
- 教室在该时段不能有课程；
- 教室在该时段不能有其他待审批或已通过预约。

提交、审批、取消和结束等操作都会写入 `room_booking_log`，因此可以查看完整操作历史。

### 6.2 学生请假

```text
学生提交
 → PENDING
 → 教务/管理员审批
 → APPROVED 或 REJECTED
```

驳回时必须填写原因。审批结果同时写入业务表和统一的 `approval_record`。

### 6.3 教师调课

```text
教师提交
 → PENDING
 → 教务/管理员重新检查冲突并审批
 → APPROVED 或 REJECTED
```

只有待审批申请可以由教师取消。审批记录同样写入 `approval_record`。

## 7. 实际课表计算

调课审批通过后，系统不会直接修改原始 `course_schedule`，而是把调课结果保存在 `schedule_change` 中。

查询某个日期的实际课表时，`services.py` 中的 `effective_schedules_for_date()` 会：

1. 根据日期查找当前有效学期；
2. 根据学期开始日期计算教学周；
3. 查出该教学周内有效的原始排课；
4. 根据单双周规则过滤课程；
5. 查询已经审批通过的换教室申请；
6. 查询该周已经审批通过的换时间或时间教室申请；
7. 合并得到实际星期、节次和教室。

因此，原始排课是基础数据，已通过的调课是覆盖层：

```text
原始 course_schedule
          +
已通过 schedule_change
          ↓
当前日期的实际课表
```

教室预约、空闲教室查询和其他冲突检查使用的也是合并后的实际课表，而不只是原始排课。

## 8. 通知逻辑

教师、教务人员和管理员可以发布通知。通知支持以下接收范围：

- `ALL_STUDENTS`：所有启用的学生；
- `CLASS`：指定班级；
- `DEPARTMENT`：指定院系；
- `USER`：指定用户。

发布通知时，系统会根据接收范围找到目标用户，并为每个用户写入一条 `notification_recipient` 记录。因此系统可以统计：

- 当前用户的未读通知数量；
- 每条通知的接收人数；
- 每条通知的已读人数；
- 每个用户对通知的阅读状态和阅读时间。

教师只能撤回自己发布且尚未过期的通知；教务人员和管理员具有全局通知管理权限。

## 9. 首页统计

`GET /api/home` 根据用户角色返回不同的首页统计：

- 学生：已选课程数、待审批教室预约数、待审批请假数；
- 教师：授课课程数、待审批教室预约数、待审批调课数；
- 教务人员和管理员：全局待审批预约、请假和调课数量；
- 所有角色：当前用户未读通知数量。

首页返回值还包含当前用户的公共资料和角色专属资料。

## 10. API 返回格式

接口统一返回以下结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

- `code = 0` 表示成功；
- 非零 `code` 表示业务或系统错误；
- HTTP 状态码仍用于区分参数错误、未登录、无权限、资源不存在、业务冲突和服务器错误。

`api_utils.py` 会递归地把数据库和后端使用的 `snake_case` 字段转换为前端使用的 `camelCase` 字段。

## 12. 核心总结

本项目的核心逻辑可以概括为：

> 以 Token 完成身份认证，以角色控制接口权限，以 SQLite 保存教学业务数据，通过教室预约、学生请假和教师调课三条状态审批流管理教学过程，并在查询和冲突检查时动态合并已经审批通过的调课结果。
